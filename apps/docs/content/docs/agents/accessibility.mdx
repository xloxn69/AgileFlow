# Accessibility Agent

The Accessibility Agent (AG-ACCESSIBILITY) is an accessibility specialist who ensures applications are usable by everyone, including people with disabilities. This agent conducts WCAG audits, tests with assistive technologies, and ensures inclusive design principles are followed.

## Capabilities

- **WCAG 2.1 Auditing**: Verify Level AA and AAA compliance
- **Screen Reader Testing**: Test with NVDA, JAWS, VoiceOver
- **Keyboard Navigation**: Ensure all functionality accessible via keyboard
- **Color Contrast Verification**: Validate ≥4.5:1 contrast ratio
- **Focus Management**: Visible focus indicators and logical tab order
- **Accessibility Testing**: Automated and manual a11y testing
- **Assistive Technology Support**: Support for screen readers, voice control, etc.
- **Inclusive Design**: Design patterns that work for all users
- **ARIA Implementation**: Semantic HTML and proper ARIA labels
- **Accessibility Documentation**: Create a11y testing guides and standards

## When to Use

Use the Accessibility Agent when:

- Designing new features or components
- Need WCAG 2.1 compliance verification
- Color contrast issues or visual accessibility concerns
- Keyboard navigation not working
- Screen reader compatibility problems
- Need to test with assistive technologies
- Creating accessibility standards
- Planning releases (accessibility testing required)
- Users with disabilities report issues
- Need ADR for accessibility decisions

## How It Works

1. **Knowledge Loading**: Agent reads expertise and WCAG standards
2. **Feature Analysis**: Agent reviews story for accessibility implications
3. **Automated Audit**: Agent runs axe, Lighthouse, WAVE
4. **Manual Testing**: Agent tests keyboard navigation, screen readers
5. **Issue Documentation**: Agent lists all accessibility issues with severity
6. **Remediation Planning**: Agent coordinates with design/UI teams
7. **Verification**: Agent re-tests after fixes
8. **Documentation**: Agent updates status.json and reports findings

## Key Behaviors

- **Inclusive by Default**: Design for all users from the start
- **Test with Real Users**: Use actual assistive technologies
- **Keyboard First**: All functionality accessible without mouse
- **Color Not Only**: Information not conveyed by color alone
- **Clear Focus**: Visible focus indicators on every interactive element
- **Semantic HTML**: Use proper HTML elements for meaning
- **No Assumptions**: Don't assume all users have keyboards or pointing devices

- Bash (run a11y tests)
- Glob (find a11y test files)
- Grep (search for a11y issues)

## WCAG 2.1 Standards

**WCAG 2.1 Levels**:
- **Level A**: Minimum accessibility
- **Level AA**: Enhanced accessibility (recommended standard)
- **Level AAA**: Optimal accessibility (aim for this)

**WCAG 2.1 Principles**:
1. **Perceivable**: Information and UI components must be perceivable
2. **Operable**: UI components and navigation must be operable
3. **Understandable**: Information and operations must be understandable
4. **Robust**: Content must be robust for assistive technologies

**Key WCAG 2.1 AA Criteria**:
- **Color Contrast**: Text ≥4.5:1, UI components ≥3:1
- **Keyboard Accessible**: All functionality via keyboard
- **No Keyboard Traps**: Can Tab out of any component
- **Focus Order**: Tab order matches visual order (logical)
- **Focus Visible**: Visible focus indicator ≥2px outline
- **Touch Target Size**: ≥44x44 CSS pixels
- **Semantic HTML**: Use proper HTML elements
- **ARIA Labels**: Buttons/links have accessible names
- **Form Labels**: All form fields properly labeled

## Accessibility Testing

**Automated Testing** (run first for quick feedback):
- **axe DevTools**: Browser extension, identifies common issues
- **Lighthouse**: Built-in audit in Chrome DevTools
- **WAVE**: WebAIM tool for visual feedback
- **Color contrast checkers**: Verify 4.5:1 ratio

**Manual Testing** (essential for real assessment):
- **Keyboard-only navigation**: Tab through entire page (no mouse)
- **Screen reader testing**: NVDA (Windows), JAWS (Windows), VoiceOver (macOS)
- **Tab order verification**: Check logical flow
- **Focus indicator visibility**: See focus on every interactive element
- **Color contrast validation**: Tool verification + visual inspection

**Screen Reader Support**:
- NVDA (Windows, free)
- JAWS (Windows, paid, most popular)
- VoiceOver (macOS/iOS, built-in)
- TalkBack (Android, built-in)

## Common Accessibility Issues

| Issue | Problem | Fix |
|-------|---------|-----|
| No labels on inputs | Screen reader doesn't know what field is | Add `<label for="id">` |
| Icon-only buttons | No name for screen readers | Add `aria-label="name"` |
| Low color contrast | Hard to read for low vision users | Increase contrast to 4.5:1 |
| Missing focus indicators | Keyboard users can't see where they are | Add visible focus outline |
| No alt text on images | Screen readers can't describe image | Add descriptive `alt="..."` |
| Keyboard traps | Users can't navigate away | Fix focus management |
| Color-only information | Color blind users miss meaning | Use icons, text, patterns |

## Accessibility Checklist

**Visual Accessibility**:
- [ ] Color contrast ≥4.5:1 for text (WCAG AA)
- [ ] Color contrast ≥3:1 for UI components
- [ ] No color-only information (use icons/text)
- [ ] Text resizable to 200% without scrolling
- [ ] Focus indicator visible (≥2px outline)
- [ ] Touch targets ≥44x44 CSS pixels
- [ ] Consistent visual design (buttons look like buttons)

**Keyboard Accessibility**:
- [ ] All functionality accessible via keyboard
- [ ] No keyboard traps (Tab can exit any component)
- [ ] Tab order logical (matches visual order)
- [ ] Enter/Space activates buttons
- [ ] Arrow keys navigate lists/menus
- [ ] Escape closes modals/popovers

**Screen Reader Accessibility**:
- [ ] Semantic HTML (button, link, form labels)
- [ ] All buttons/links have accessible name
- [ ] Images have alt text (or hidden if decorative)
- [ ] Form fields have labels (or aria-label)
- [ ] Form errors announced to screen readers
- [ ] Landmarks present (main, nav, complementary)
- [ ] Skip links for keyboard navigation
- [ ] Live regions for dynamic content

## Inclusive Design Patterns

**Forms**:
```html
<!-- Good: Label associated with input -->
<label for="email">Email</label>
<input id="email" type="email" required aria-required="true">
<span class="error" role="alert">Email is required</span>

<!-- Bad: Placeholder only, no label -->
<input placeholder="Email" type="email">
```

**Buttons**:
```html
<!-- Good: Clear accessible name -->
<button aria-label="Close menu">
  <svg aria-hidden="true"><!-- icon --></svg>
</button>

<!-- Bad: Icon-only with no label -->
<button>
  <svg><!-- icon --></svg>
</button>
```

**Images**:
```html
<!-- Good: Descriptive alt text -->
<img src="profile.jpg" alt="Sarah Chen, product manager">

<!-- Bad: Alt text missing -->
<img src="profile.jpg">

<!-- Good: Decorative image hidden -->
<img src="decoration.jpg" alt="" aria-hidden="true">
```

**Color Contrast**:
```text
Good: White text (#FFF) on blue (#0066CC) = 8.6:1 ✅ AAA
Bad:  White text (#FFF) on gray (#999) = 3.9:1 ❌ AA (need 4.5:1)
Good: White text (#FFF) on black (#000) = 21:1 ✅ AAA
```

## Remediation Priority

| Priority | Severity | Examples |
|----------|----------|----------|
| Critical | Blocks core functionality | No keyboard access to features, missing form labels |
| Major | Significantly impacts usability | Low color contrast, missing focus indicators |
| Minor | Impacts some users | Missing ARIA annotations, verbose error messages |

## Motion & Animation

**Accessibility considerations**:
- [ ] Animations respect `prefers-reduced-motion`
- [ ] No seizure-inducing flashing (no >3 flashes/second)
- [ ] Autoplay disabled by default
- [ ] Users can pause animations

## Accessibility Principles

- **Inclusive by design**: Consider all users from the start
- **Test with real users**: Use actual assistive technologies
- **No shortcuts**: WCAG compliance is mandatory, not optional
- **Layers of accessibility**: Visual, keyboard, screen reader, cognitive
- **Continuous improvement**: Fix issues as discovered
- **Education**: Help team understand why accessibility matters
