# /agileflow:skill:test

Test a skill to verify it activates correctly and produces expected results.

## Quick Start

```bash
# Interactive mode - select which skill to test
/agileflow:skill:test

# Test a specific skill
/agileflow:skill:test supabase-swift
```

## How It Works

This command validates that a skill:

1. **Has correct structure** - All required files present
2. **Has valid metadata** - Frontmatter is properly formatted
3. **Activates correctly** - Triggers on appropriate prompts
4. **Produces expected output** - Returns properly formatted results

## Validation Checks

### Structure Checks

| Check | Pass Condition | Example |
|-------|----------------|---------|
| SKILL.md exists | File present in skill directory | `.claude/skills/supabase-swift/SKILL.md` |
| Frontmatter valid | Has `name:` and `description:` | Both required fields present |
| references.md | File exists (recommended) | `.claude/skills/supabase-swift/references.md` |
| cookbook/ | Directory exists if referenced | Files match SKILL.md references |
| .mcp.json | Valid JSON if present | Parses without errors |

### Content Checks

| Check | Pass Condition | Notes |
|-------|----------------|-------|
| When to Use | Section present with activation triggers | Describes when skill is useful |
| Description length | Under 1024 characters | Brief but complete description |
| SKILL.md size | Under 500 lines | Keeps skills maintainable |
| Cookbook references | All referenced files exist | No broken links |
| No broken links | All local file references valid | Points to actual files |

### MCP Checks (if .mcp.json exists)

| Check | Pass Condition | Notes |
|-------|----------------|-------|
| Valid JSON | Parses without errors | Proper JSON syntax |
| Has mcpServers | Contains mcpServers object | Root-level mcpServers key |
| Command exists | Command is npx or valid executable | Runnable command path |
| Env vars documented | Any ${VAR} has comments | Explains environment variables |

## When to Test Skills

Test skills when:

- **Creating a new skill** - Verify it's correctly structured
- **Editing a skill** - Ensure changes didn't break anything
- **Before sharing** - Validate before sharing with team
- **After major changes** - Test after significant edits
- **Periodic maintenance** - Regular validation of installed skills
- **Troubleshooting** - When skill doesn't seem to work

## Best Practices

1. **Test after creation** - New skills should always be tested
2. **Test after editing** - Verify changes don't break functionality
3. **Test before sharing** - Validate with team members
4. **Keep tests documented** - Note what was tested and results
5. **Fix issues immediately** - Don't use broken skills
6. **Archive if failing** - Move unusable skills to backup

## Using Tested Skills

After a successful test:

1. **Use in projects** - Skill is ready for development
2. **Share with team** - Copy skill to other projects
3. **Document locally** - Link to in project documentation
4. **Version control** - Commit if skills are tracked
5. **Monitor updates** - Retest after major changes
